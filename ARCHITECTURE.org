* Overview

This repository manages all of Ray's system configurations — NixOS desktops,
a nix-darwin MacBook, and standalone Home Manager profiles — using the
*Dendritic pattern*.

The core idea: the ~flake.nix~ is a thin entry point that does nothing except
wire up inputs and hand off to a collection of modules. Every piece of real
logic lives in a file under ~./modules/~, and each of those files is a
*flake-parts module*.

** Key properties

- No ~specialArgs~ / ~extraSpecialArgs~ pass-thru for shared values. Instead,
  shared values (username, email) are declared as top-level flake-parts
  options and read via ~config.*~.
- ~import-tree~ auto-imports every ~.nix~ file under ~./modules/~ — no
  manual list of imports to maintain in the flake.
- NixOS system-level modules (not flake-parts modules) live under
  ~modules/_nixos/~ and ~modules/_hardware/~. The leading underscore causes
  ~import-tree~ to skip those subdirectories, so they are never mistakenly
  loaded as flake-parts modules.
- Hosts reference the shared Home Manager config via
  ~flake.modules.homeManager.base~ — a registry provided by
  ~flake-parts.flakeModules.modules~ — rather than by passing a path or a
  ~specialArgs~ value around.

* How it all fits together

** The entry point (~flake.nix~)

#+begin_src nix
outputs = inputs:
  inputs.flake-parts.lib.mkFlake { inherit inputs; }
  (inputs.import-tree ./modules);
#+end_src

~import-tree ./modules~ returns a single merged flake-parts module that is the
union of every ~.nix~ file it finds under ~./modules/~. ~mkFlake~ receives that
merged module and evaluates it with the standard flake-parts machinery.

Because ~import-tree~ skips any path component that starts with ~_~, the files
under ~_nixos/~ and ~_hardware/~ are never touched here — they are only
imported explicitly from host files.

** Shared metadata (~modules/meta.nix~)

Declares two top-level flake-parts options:

| Option     | Default              | Purpose                         |
|------------+----------------------+---------------------------------|
| ~username~ | ~"ray"~              | Primary Unix username           |
| ~email~    | ~"kjwdev01@gmail.com"~ | Git / jujutsu author email    |

Any flake-parts module in the tree can read these as ~config.username~ and
~config.email~ without needing them threaded through ~specialArgs~.

Note: these options live at the flake-parts level and are *not* accessible
from inside Home Manager ~deferredModule~ bodies, which have their own
~config~ scope.

** Supported systems (~modules/systems.nix~)

#+begin_src nix
systems = [ "aarch64-darwin" "x86_64-linux" "aarch64-linux" ];
perSystem = { pkgs, ... }: { formatter = pkgs.alejandra; };
#+end_src

This is the flake-parts ~systems~ option, which controls what
~perSystem~ outputs are generated for. It also sets the project-wide
formatter (~nix fmt~ uses ~alejandra~).

** The modules registry (~modules/flake-parts.nix~)

#+begin_src nix
imports = [ inputs.flake-parts.flakeModules.modules ];
#+end_src

Enabling this built-in flake-parts module unlocks the ~flake.modules.*~ option
namespace, which can hold ~deferredModule~ values. The Home Manager config
is stored as ~flake.modules.homeManager.base~ so hosts can reference it
with a single attribute path rather than importing a file path.

** Home Manager config distribution

Unlike a monolithic HM config file, the Home Manager configuration is
*distributed* across many flake-parts modules. Each module under ~shell/~,
~tools/~, ~packages/~, and ~theming/~ contributes to
~flake.modules.homeManager.base~ by adding its own ~deferredModule~ fragment.
Flake-parts merges all contributions automatically.

The base identity (~home-base.nix~) sets username, homeDirectory, and
stateVersion. Standalone entries (~home-configurations.nix~) register
linux/darwin/termux configurations that all consume ~base~.

** Builder modules

Each builder module follows the same pattern:

1. Declare a flake-parts option (e.g. ~configurations.nixos~) whose value is
   an attribute set of ~{ module = deferredModule; }~ records.
2. In ~config~, map over that attribute set and call the appropriate builder
   (~nixpkgs.lib.nixosSystem~, ~darwin.lib.darwinSystem~, or
   ~home-manager.lib.homeManagerConfiguration~) to produce the final
   ~flake.*Configurations~ output.

| File              | Option declared          | Output produced              |
|-------------------+--------------------------+------------------------------|
| ~nixos.nix~       | ~configurations.nixos~   | ~flake.nixosConfigurations~  |
| ~darwin.nix~      | ~configurations.darwin~  | ~flake.darwinConfigurations~ |
| ~home.nix~        | ~configurations.home~    | ~flake.homeConfigurations~   |

** Host files (~modules/hosts/~)

Each host file is a flake-parts module that sets one entry on the relevant
~configurations.*~ option. The value it provides is a ~deferredModule~:
a NixOS / Darwin module that is not evaluated here but handed to the builder.

Example (~modules/hosts/nixos.nix~, abbreviated):

#+begin_src nix
{ config, inputs, ... }: {
  configurations.nixos.nixos.module = {
    imports = [
      ../_hardware/nixos.nix      # hardware-configuration.nix for this machine
      ../_nixos/options.nix       # my.profile / my.stacks.* options
      ../_nixos/core/users.nix
      # … more NixOS system modules …
      inputs.home-manager.nixosModules.home-manager
    ];

    networking.hostName = "nixos";
    # … machine-specific NixOS settings …

    home-manager.users.ray = config.flake.modules.homeManager.base;
  };
}
#+end_src

The last line is the key integration point: instead of passing a file path or
using ~specialArgs~, the host looks up the shared HM config from the registry.

* The ~_~ convention

~import-tree~ skips any path whose name begins with ~_~. This is used to keep
NixOS system-level modules out of the flake-parts evaluation:

- ~modules/_nixos/~ — standard NixOS modules (~options~, ~config~ keys, using
  ~pkgs~, ~lib~, ~modulesPath~, etc.). These are valid NixOS module system
  files but are *not* flake-parts modules. They are imported explicitly from
  host ~deferredModule~ definitions.
- ~modules/_hardware/~ — per-machine hardware configuration files, also
  NixOS modules, imported the same way.

* NixOS stacks (~_nixos/stacks/~)

Stacks are optional feature bundles gated behind ~my.stacks.<name>.enable~
options (declared in ~_nixos/options.nix~). Each stack uses ~lib.mkIf~ to
conditionally include its packages and services.

| Stack     | Gate                         | Packages                              |
|-----------+------------------------------+---------------------------------------|
| ~gaming~  | ~my.stacks.gaming.enable~    | Steam, Lutris, MangoHud, Gamemode     |
| ~ai~      | ~my.stacks.ai.enable~        | Python3, PyTorch, NumPy               |
| ~music~   | ~my.stacks.music.enable~     | Audacity, Ardour                      |

* Adding a new host

1. Create ~modules/hosts/<name>.nix~.
2. Set ~configurations.nixos.<name>.module~ (or ~darwin~) to a deferredModule.
3. Import the right hardware file from ~../_hardware/<name>.nix~ (create it if
   needed) and the shared ~_nixos/~ modules you need.
4. Wire up Home Manager with:
   #+begin_src nix
   home-manager.users.ray = config.flake.modules.homeManager.base;
   #+end_src
5. No changes to ~flake.nix~ — ~import-tree~ picks up the new file
   automatically.

* Adding a new NixOS system module

Drop a ~.nix~ file anywhere under ~modules/_nixos/~. It will be ignored by
~import-tree~ and must be explicitly imported from whichever host needs it.

* Adding a new Home Manager tool/program

Create a file under ~modules/tools/~ or ~modules/shell/~. The file should
contribute to ~flake.modules.homeManager.base~ as a ~deferredModule~:

#+begin_src nix
{ ... }: {
  flake.modules.homeManager.base = { pkgs, ... }: {
    programs.mytool = {
      enable = true;
      # … config …
    };
  };
}
#+end_src

~import-tree~ will pick it up automatically — no manual registration needed.
